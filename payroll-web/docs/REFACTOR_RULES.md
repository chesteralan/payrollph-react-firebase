# Refactoring & Code Quality — payroll-web

_Generated 2026-05-25 — v0.0.0_
This guide sets quality rules for clean coding, codebase maintenance, and deprecation protocols in the **payroll-web** repository.

---

## 📐 Quality Standards

- **Single Responsibility Principle**: A file should do one thing. If a component grows past 200 lines, extract helper functions or subcomponents.
- **Strict Typing**: Avoid `any` declarations. Use detailed interface structures or utility types.

---

## 🛠️ Refactoring Trigger Checklist

Refactor modules when you encounter these code smells:

| Smell                 | Detection Criteria                                  | Remedy                                      |
| :-------------------- | :-------------------------------------------------- | :------------------------------------------ |
| **Deep Nesting**      | If statement nesting goes past 3 levels.            | Guard clauses / early returns.              |
| **Duplicate Logic**   | Similar logic exists in 3 or more files.            | Extract to custom hooks or helper modules.  |
| **Fragile Functions** | Functions requiring multiple parameter adjustments. | Object parameters / option config payloads. |

---

## ⚠️ Deprecation Protocol

> [!CAUTION]
> Avoid making breaking changes directly on active branches:
>
> 1. **Mark as Deprecated**: Use JSDoc `@deprecated` tag to flag methods.
> 2. **Add Warn logs**: Output runtime console warnings when invoked in development.
> 3. **Schedule Removal**: Set a future milestone version to purge obsolete methods.
