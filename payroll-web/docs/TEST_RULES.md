## 🧪 Testing Stack

- **Framework**: `Vitest`
- **Runner**: `vitest`
- **Coverage Target**: 80%+ for business logic, 60%+ overall

---

## 📋 Test Types

### Unit Tests

- One test file per module, co-located or in `__tests__/`
- Mock external dependencies (API calls, DB, browser APIs)
- Test public interface, not implementation details

### Integration Tests

- Cover critical user flows end-to-end through the backend
- Use a test database or in-memory substitute
- Verify error paths and edge cases

### E2E Tests (if applicable)

- Cover critical user journeys
- Run against a staging-like environment
- Keep test count focused — quality over quantity

## ✅ Naming Conventions

| Type               | Pattern                            | Example                                                 |
| :----------------- | :--------------------------------- | :------------------------------------------------------ |
| **Unit test file** | `*.test.ts` or `*.spec.ts`         | `auth.service.test.ts`                                  |
| **Test suite**     | `describe('ModuleName', ...)`      | `describe('AuthService', ...)`                          |
| **Test case**      | `test('should ... when ...', ...)` | `test('should reject invalid token when expired', ...)` |

---

## 🚫 Testing Anti-Patterns

> [!WARNING]
>
> - **Don't test internal implementation**: Test behavior, not private methods.
> - **Don't share mutable state between tests**: Each test must be isolated.
> - **Don't use `test.only` / `describe.only`**: Never commit focused tests.
> - **Don't mock what you don't own**: Mock your abstraction layer, not third-party SDKs directly.
